## What This Project Is

This is a **Claude-powered enterprise agent platform** built on top of [microsoft-foundry/foundry-agent-webapp](https://github.com/microsoft-foundry/foundry-agent-webapp) as a base. It is a production-ready web application that connects users to Azure AI Foundry agents through a real-time streaming chat interface, secured by Microsoft Entra ID authentication and deployed to Azure Container Apps.

Key flow: `React UI вҶ’ MSAL token вҶ’ POST /api/chat/stream вҶ’ AI Foundry Agent вҶ’ SSE chunks вҶ’ UI`

## Mock Mode вҖ” Frontend-Only Development

**Use this when**: working on UI components, styling, state management, or any frontend feature that doesn't require a real agent response. No Azure account, no `.env` files, no backend needed.

### Enabling Mock Mode

`frontend/.env.local` already contains the toggle:

```ini
VITE_MOCK_MODE=true
```

That single line is all that's needed. Start the frontend normally:

```bash
cd frontend
npm install
npm run dev
# вҶ’ http://localhost:5173
```

### What Mock Mode Does

| Layer | Real Mode | Mock Mode |
|-------|-----------|-----------|
| Auth | MSAL PKCE вҶ’ Azure AD redirect | Stub user auto-authenticated, no redirect |
| Token | `acquireTokenSilent` / popup | Returns `"mock-token"` immediately |
| API | `POST /api/chat/stream` вҶ’ ASP.NET Core вҶ’ Azure AI Foundry | `MockChatService` simulates word-by-word streaming locally |
| Backend | Required (port 8080) | Not required |
| `.env` files | Required | Not required |

The simulated stream cycles through 3 canned responses that demonstrate markdown rendering, code blocks, and tables. The Stop button, message queue, New Chat, Regenerate, and Edit all work normally вҖ” `MockChatService` dispatches the same reducer actions as the real service.

### How It Works (Implementation)

The mock mode path in `main.tsx`:

```
MsalProvider (placeholder creds вҖ” never used for real tokens)
  в””в”Җв”Җ MockAppProvider  (immediately dispatches AUTH_INITIALIZED with stub user)
        в””в”Җв”Җ ThemeProvider
              в””в”Җв”Җ MockApp  (renders AgentChat directly, skips useMsalAuthentication)
                    в””в”Җв”Җ AgentChat  (uses MockChatService instead of ChatService)
```

Key files:
- `frontend/src/mock/MockApp.tsx` вҖ” renders `AgentChat` without auth boilerplate
- `frontend/src/mock/mockChatService.ts` вҖ” `MockChatService extends ChatService`, overrides `sendMessage` / `cancelStream` / conversation methods
- `frontend/src/contexts/AppContext.tsx` вҶ’ `MockAppProvider` вҖ” no `useMsal()`, immediately sets `auth.status = 'authenticated'`
- `frontend/src/hooks/useAuth.ts` вҖ” returns `"mock-token"` when `VITE_MOCK_MODE=true`
- `frontend/plugins/envcheck.ts` вҖ” skips missing-env-var check when `VITE_MOCK_MODE=true`

### Disabling Mock Mode

To connect a real Azure AI Foundry agent:

```bash
# Option 1: remove the file
rm frontend/.env.local

# Option 2: set the flag to false in frontend/.env.local
# VITE_MOCK_MODE=false

# Then provision Azure resources (generates real .env files)
azd up
```

## How the Auth Flow Works

### Frontend вҶ’ Backend

1. Browser loads the SPA; MSAL.js initializes and checks the cache
2. If no cached session вҶ’ `unauthenticated` state вҶ’ login redirect
3. MSAL completes PKCE flow вҶ’ `authenticated` state, `AccountInfo` stored
4. Every API call: `acquireTokenSilent()` first, fallback to `acquireTokenPopup()`
5. Token is sent as `Authorization: Bearer <JWT>` on every request to `/api/*`

### Backend JWT Validation

`Program.cs` uses `AddMicrosoftIdentityWebApi` to validate tokens. Dual audience support:

```csharp
options.TokenValidationParameters.ValidAudiences = new[]
{
    builder.Configuration["AzureAd:ClientId"],
    $"api://{builder.Configuration["AzureAd:ClientId"]}"
};
```

All endpoints except `/api/health` require the `Chat.ReadWrite` scope via `.RequireAuthorization("RequireChatScope")`.

### Backend вҶ’ AI Foundry (Credential Strategy)

```csharp
// Development: ChainedTokenCredential (az login / azd auth login)
// Production: ManagedIdentityCredential (user-assigned MI)
// OBO (opt-in): OnBehalfOfCredential via ManagedIdentityClientAssertion (FIC)
```

- **Default (production)**: User-assigned managed identity with `Cognitive Services OpenAI Contributor` + `Azure AI Developer` roles on the AI Foundry resource
- **OBO (opt-in)**: Enable with `azd env set ENABLE_OBO true && azd up`. Passes the user's identity to the Agent Service API. Requires admin consent to **Azure Machine Learning Services** (appId `18a66f5f-...`) вҖ” NOT Microsoft Cognitive Services
- **Local dev**: Always uses `ChainedTokenCredential(AzureCliCredential, AzureDeveloperCliCredential)` regardless of OBO setting

## How the Agent Connection Works

The `AgentFrameworkService` uses a **lazy singleton** pattern:

1. On first request, `GetAgentAsync()` acquires a `SemaphoreSlim` lock
2. Calls `aiProjectClient.GetAIAgentAsync(name: agentId)` via the Agent Framework extension
3. Caches the result in `s_cachedAgent` (static, shared across requests)
4. Subsequent requests reuse the cached agent; failures retry on the next request

**Hybrid SDK approach**: Agent Framework (`Microsoft.Agents.AI.AzureAI`) is used for agent loading and metadata. Direct `ProjectResponsesClient` is used for streaming вҖ” because the `IChatClient` abstraction doesn't expose `McpToolCallApprovalRequestItem`, `FileSearchCallResponseItem`, or `OutputTextAnnotations`.

```csharp
// Loading (Agent Framework)
ChatClientAgent agent = await projectClient.GetAIAgentAsync(name: agentId, ct);

// Streaming (Direct SDK вҖ” required for MCP + annotations)
ProjectResponsesClient responsesClient = projectClient.OpenAI
    .GetProjectResponsesClientForAgent(new AgentReference(agentId), conversationId);
```

## The Streaming Architecture

### Backend SSE Pipeline

`POST /api/chat/stream` вҶ’ sets SSE headers вҶ’ creates/reuses conversation вҶ’ streams `IAsyncEnumerable<StreamChunk>` from `AgentFrameworkService.StreamMessageAsync()` вҶ’ writes `data: {...}\n\n` per event вҶ’ flushes after each write

**SSE Event Sequence** (always in this order):

| # | Event Type | Payload | When |
|---|-----------|---------|------|
| 1 | `conversationId` | `{conversationId: string}` | Always first |
| 2 | `chunk` | `{content: string}` | 0-N text deltas |
| 3 | `annotations` | `{annotations: [...]}` | 0-N after item complete |
| 4 | `mcpApprovalRequest` | `{approvalRequest: {...}}` | 0-1, pauses stream |
| 5 | `toolUse` | `{toolName: string}` | When agent uses a tool |
| 6 | `usage` | `{duration, promptTokens, completionTokens, totalTokens}` | Before done |
| 7 | `done` | `{}` | Always last |
| вҖ” | `error` | `{message: string}` | On exception |

### Frontend State Machine

Chat status transitions:

```
idle в”Җв”ҖCHAT_SEND_MESSAGEв”Җв”Җв–ә sending в”Җв”ҖCHAT_START_STREAMв”Җв”Җв–ә streaming
  в–І                                                             в”Ӯ
  в””в”Җв”ҖCHAT_STREAM_COMPLETE / CHAT_CANCEL_STREAM / CHAT_MCP_APPROVAL_REQUEST
  в””в”Җв”ҖCHAT_CLEAR_ERRORв”Җв”Җв”Җ error в—„в”Җв”Җв”Җв”Җв”Җв”ҖCHAT_ERRORв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

**SSE вҶ’ Reducer Action Mapping** (in `frontend/src/services/chatService.ts`):

| SSE Event | Action | Effect |
|-----------|--------|--------|
| `conversationId` | `CHAT_START_STREAM` | Set conversationId, status вҶ’ streaming |
| `chunk` | `CHAT_STREAM_CHUNK` | Append content to streaming message |
| `annotations` | `CHAT_STREAM_ANNOTATIONS` | Add citations to message |
| `mcpApprovalRequest` | `CHAT_MCP_APPROVAL_REQUEST` | Show approval UI, status вҶ’ idle |
| `usage` | `CHAT_STREAM_COMPLETE` | Add token counts, status вҶ’ idle |
| `toolUse` | `CHAT_STREAM_TOOL_USE` | Set activeToolUse indicator |
| `error` | `CHAT_ERROR` | Set error state |

**Message Queue**: While streaming, new messages queue in `pendingMessages[]` and auto-send when the stream completes.

**Stream Retry**: Automatic retry up to 3Г— with exponential backoff. On exhaustion, `CHAT_RECOVER_MESSAGE` removes the failed messages and restores the user's text to the input.

**Performance**: `useDeferredValue(messages)` keeps input responsive during rapid streaming вҖ” immediate `messages` drives scroll/a11y, deferred drives the heavy message list render.

## Coding Conventions

### TypeScript / React

- Strict TypeScript: `noImplicitAny`, `strictNullChecks` вҖ” no `any` types
- Functional components with typed props interfaces
- `useState` for local state; `AppContext` + `useReducer` for shared state
- MSAL: always `acquireTokenSilent()` first, then `acquireTokenPopup()` fallback
- Environment variables: access at **module level only** (Vite build-time replacement)
- Memoize: `useMemo` for computed values, `useCallback` for stable references in hooks
- Never store tokens in component state
- Run `npm install` from `frontend/` directory (`.npmrc` sets `legacy-peer-deps`)
- Verify `npm ci` before committing package changes

### C# / ASP.NET Core

- All I/O must be `async/await` with `CancellationToken` вҖ” never `.Result` or `.Wait()`
- Pass `CancellationToken` through the entire call chain
- Use `IAsyncEnumerable<T>` with `[EnumeratorCancellation]` for streaming
- All endpoints need `.RequireAuthorization("RequireChatScope")`
- Error responses use RFC 7807 Problem Details (`ErrorResponseFactory.CreateFromException()`)
- Services implementing `IDisposable`: use `ObjectDisposedException.ThrowIf` guards in all public methods
- Credential strategy: `ChainedTokenCredential` (dev) vs `ManagedIdentityCredential` (prod) вҖ” never `DefaultAzureCredential`

### Bicep / Infrastructure

- Resource names use `uniqueString()` token: `'${abbrs.appContainerApps}web-${token}'`
- All parameters need `@description()` decorator
- RBAC: use `guid()` for assignment names + specify `principalType: 'ServicePrincipal'`
- User-assigned MI for ACR pull (not admin credentials)
- Container Apps: scale-to-zero (`minReplicas: 0`), HTTPS-only, `targetPort: 8080`

### Git Commits

Commits are gated by a `preToolUse` hook вҖ” **do not run `git commit` directly**. Follow the `committing-code` skill:

1. `git diff --staged` to review changes
2. Write message to `COMMIT_MESSAGE.md`
3. `git commit -F COMMIT_MESSAGE.md`
4. Delete `COMMIT_MESSAGE.md`

Conventional commit format: `<type>(<scope>): <subject>` with `feat`, `fix`, `refactor`, `chore`, `docs` types.
